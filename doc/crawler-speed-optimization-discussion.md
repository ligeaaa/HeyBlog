# 爬虫速度优化讨论提纲

## 结论先行

这轮讨论只看一件事：**怎样让 HeyBlog 抓博客更快**。

目前最重要的前提有两个：

1. 候选页并发抓取已经做过一部分，不应再把它当成“未开始”的优化项。
2. 现阶段还没有阶段耗时和请求分布基线，所以不能直接把某一个环节写成确定主瓶颈。

## 现状判断

- 速度方面已经完成的部分：
  - 单个 blog 内的候选页已经支持并发抓取。
  - 并发抓取后的“按候选顺序处理”和“并发度为 1 的兼容回退”已有测试保护。

- 速度方面还未解决的部分：
  - blog 级调度仍然是串行的。
  - 后台持续运行模式仍以 `max_nodes=1` 为粒度推进。
  - runtime 和日志还缺少阶段耗时、请求数、候选页来源分布等观测，因此我们只能提出“重点怀疑对象”，不能直接下结论。

## 可讨论的两条主线

### 主线一：先减少总工作量

建议优先讨论：

1. 是否应该给候选页增加优先级和预算，而不是一视同仁地都抓。
2. 是否应该更早结束“低价值 blog”的处理，减少无效等待。
3. 是否应该先补速度观测，再决定真正要投在哪个优化点上。

### 主线二：再考虑更大调度改造

建议作为第二阶段讨论：

1. blog 级串行是不是已经成为主瓶颈。
2. runtime 的单 active blog 状态模型是否需要扩展。
3. 多 blog 并发是否值得引入，以及它会带来哪些新的复杂度。

## 推荐路线

推荐先走“证据驱动的分阶段路线”，而不是一开始就冲向多 blog 并发：

1. Phase 0：先补速度基线
   - 单 blog 总耗时
   - 首页抓取耗时
   - 候选页抓取耗时
   - 候选页发现耗时
   - 候选页处理/入库耗时
   - 候选页数量与来源分布
   - 请求成功/失败分布
   - 吞吐口径固定为“单位 wall-clock 时间内完成的 blog 数”和“单 blog 平均 wall-clock 耗时”
   - 这一步是后续进入实现讨论的前置门槛，不是可选建议。

2. Phase 1：先做低风险速度优化
   - 候选页预算
   - 候选页优先级
   - 更早停止无价值处理
   - 导出/收尾阶段是否需要减频或异步化
   - 这一阶段只在现有单 active blog 模型内做控制面优化，不默认打开多 blog 并发。
   - 导出/收尾先保留在观测清单里，不默认视为最高优先级优化点。

3. Phase 2：只在证据支持时再升级
   - 如果基线显示 blog 级串行占主导，再进入多 blog 并发设计。
   - 如果 Phase 1 的收益已经足够，则不急着打开更复杂的调度改造。

建议在 Phase 1 开始前先约定一个最小可接受收益阈值：

- 主指标必须出现稳定正向提升。
- 或单 blog 平均耗时必须下降到足以支撑继续投入。
- 如果收益低于门槛，就不继续堆更多控制面复杂度。

## 推荐讨论顺序

1. 先确认当前速度基线是否存在。
2. 再讨论“减少总工作量”还是“提高并发层级”哪个更值得先做。
3. 如果没有观测数据，不建议直接进入实现讨论。
