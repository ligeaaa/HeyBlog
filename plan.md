# HeyBlog 计划文档

## 项目概述

HeyBlog 是一个面向博客友链网络的爬虫系统。程序从一组初始博客网址出发，自动发现博客中的友链页面和友链列表，持续扩展，最终生成博客连接图。

第一阶段目标是实现一个可运行、可中断恢复、可观测、可导出图数据的 MVP。当前优先保证抓取链路稳定、状态可恢复、数据结构清晰，不在第一阶段追求极致识别率或复杂前端功能。

---

# 1. 持久化

## 1.1 目标

持久化模块负责保存系统运行过程中的所有关键状态，使程序具备以下能力：

- 中断后可恢复
- 节点去重
- 连接关系持久保存
- 支持统计与可视化查询
- 支持后续导出图数据

## 1.2 存储策略

建议采用分层存储：

- `seed.csv`：仅作为初始种子导入文件
- `SQLite`：作为运行时主存储
- `JSON / CSV`：作为导出格式

不建议长期使用 `csv + json` 作为主存储，因为运行时会涉及：

- 去重
- 状态更新
- 边关系管理
- 失败重试
- 条件查询
- 中断恢复

这些都更适合交给数据库完成。

## 1.3 核心数据模型

### blogs

用于存储博客节点。

建议字段：

- `id`
- `url`
- `normalized_url`
- `domain`
- `status_code`
- `crawl_status`
- `friend_links_count`
- `depth`
- `source_blog_id`
- `last_crawled_at`
- `created_at`
- `updated_at`

建议状态：

- `WAITING`
- `PROCESSING`
- `FINISHED`
- `FAILED`
- `SKIPPED`

### edges

用于存储博客之间的连接关系。

建议字段：

- `id`
- `from_blog_id`
- `to_blog_id`
- `link_url_raw`
- `link_text`
- `discovered_at`

### crawl_logs

用于记录爬取过程日志，便于调试、恢复和前端展示。

建议字段：

- `id`
- `blog_id`
- `stage`
- `result`
- `message`
- `created_at`

## 1.4 URL 去重与规范化

所有链接在入库前都需要做规范化处理，用于避免重复节点。

建议规则：

- 去掉 fragment
- 去掉常见追踪参数
- 统一尾部 `/`
- 尽量统一 scheme
- 提取 domain
- 在合适情况下归一到站点级 URL

需要注意保留部分子路径博客场景，例如：

- `https://example.com/blog/`

## 1.5 中断恢复

中断恢复依赖数据库中的任务状态，而不是 Docker 本身。

恢复机制建议：

- 启动时扫描未完成任务
- 上次异常终止时处于 `PROCESSING` 的任务重新标记为可重试
- 按状态继续调度，不重复处理已完成节点

## 1.6 导出结果

第一阶段建议支持以下导出格式：

- `nodes.csv`
- `edges.csv`
- `graph.json`

这些导出结果既可用于分析，也可供前端直接消费。

# 2. 前端

## 2.1 目标

前端负责展示系统运行状态与爬取结果，重点是“可观测性”和“图谱预览”，而不是复杂交互。

第一阶段建议做一个轻量控制台式面板，便于观察程序当前执行情况。

## 2.2 第一阶段功能

### 运行进度可视化

展示当前爬虫的整体进度，例如：

- 当前任务总数
- 已完成任务数
- 正在处理任务数
- 等待任务数
- 失败任务数

### 站点处理状态展示

展示站点列表与当前状态，例如：

- 哪些网站正在处理
- 哪些网站已完成
- 哪些网站失败
- 每个网站最后处理时间
- 每个网站状态码
- 每个网站发现的友链数量

### 基础统计信息

展示关键统计指标，例如：

- 已发现博客总数
- 已建立边总数
- 当前队列长度
- 平均每个博客发现的友链数
- 失败率
- 最大抓取深度

### 图谱可视化

第一阶段可做简化版图谱展示：

- 节点表示博客
- 边表示友链关系
- 支持基础缩放和平移
- 支持按节点状态或深度着色

## 2.3 前端数据来源

前端不直接访问数据库，而是通过后端 API 获取：

- 运行状态
- 节点列表
- 边列表
- 统计信息
- 日志摘要

## 2.4 前端阶段划分

建议分两步：

1. 第一版先做状态面板和统计卡片
2. 第二版再做交互式图谱可视化

原因是图谱展示复杂度更高，但不是 MVP 主链路阻塞项。

# 3. 后端

## 3.1 目标

后端负责整个系统的核心能力，包括：

- 导入种子网址
- 抓取网页
- 发现友链页
- 提取和过滤友链
- 写入持久化存储
- 为前端提供查询接口

## 3.2 核心流程

整体采用类似 BFS 的调度方式：

1. 从 `seed.csv` 导入初始节点
2. 从待处理队列中取出一个博客
3. 请求博客首页
4. 寻找友链入口
5. 进入候选友链页
6. 提取页面中的候选链接
7. 做过滤、规范化和去重
8. 将新节点写入数据库
9. 将边关系写入数据库
10. 更新当前博客状态
11. 重复执行直到达到上限或队列为空

## 3.3 友链发现策略

建议拆为两个阶段。

### 阶段一：发现友链页

优先从以下区域查找：

- 顶部导航
- 侧边栏
- 页脚
- 固定页面入口

关键词候选：

- `友链`
- `友情链接`
- `Links`
- `Blogroll`
- `Friends`
- `邻居`
- `小伙伴`

路径候选：

- `/links`
- `/friends`
- `/friend-links`
- `/blogroll`

### 阶段二：提取友链列表

从候选友链页中抽取外部链接，并进入过滤与判定流程。

## 3.4 链接过滤规则

### 域名过滤

- 排除当前博客自身域名
- 默认只保留外链

### 平台过滤

排除非博客平台，例如：

- GitHub
- Twitter / X
- 知乎
- 微博
- Bilibili
- YouTube
- Telegram

### 黑名单过滤

排除无效链接，例如：

- 广告联盟
- 备案查询
- 统计脚本相关页面
- 登录注册页
- 联系方式页面

### 博客候选判断

第一阶段只做轻量规则判断，不追求完全准确。后续可扩展为博客置信度评分。

## 3.5 调度与限流

建议支持以下调度参数：

- `max_nodes`
- `max_depth`
- `max_outgoing_links_per_blog`
- `max_retries`
- `crawl_interval_hours`

建议支持以下抓取约束：

- 请求超时
- 重试机制
- 限速
- 单域名请求频率限制
- 自定义 `User-Agent`

## 3.6 网页获取策略

建议采用双模式：

### 默认模式

直接通过 HTTP 获取 HTML。

适用于：

- 静态博客
- 大部分常见博客框架

### 兜底模式

在必要时使用浏览器或 MCP 访问页面。

适用于：

- 动态渲染站点
- 首页结构复杂、静态抓取效果差的网站

第一阶段建议默认使用 HTTP 模式，浏览器 / MCP 作为增强能力预留。

## 3.7 技术选型建议

建议第一阶段使用：

- Python
- `httpx`
- `BeautifulSoup4` 或 `lxml`
- `SQLite`
- `FastAPI`
- `Docker + docker compose`

`FastAPI` 的作用：

- 提供前端需要的接口
- 提供运行状态查询
- 提供图谱数据接口
- 为后续控制接口预留扩展空间

# 4. API 文档

## 4.1 目标

API 层负责为前端和外部调用方提供统一数据接口。第一阶段以“读接口”为主，必要时补充少量控制接口。

建议接口风格：

- REST API
- 使用 OpenAPI 自动生成文档

## 4.2 第一阶段建议接口

### 系统状态接口

用于查询爬虫总体运行状态。

建议返回：

- 当前是否运行中
- 待处理任务数
- 正在处理任务数
- 已完成任务数
- 失败任务数
- 已发现博客总数
- 已建立边总数

示例路径：

- `GET /api/status`

### 节点列表接口

用于获取博客节点列表。

支持字段：

- `id`
- `url`
- `domain`
- `crawl_status`
- `status_code`
- `friend_links_count`
- `depth`
- `updated_at`

示例路径：

- `GET /api/blogs`

可选支持：

- 分页
- 按状态过滤
- 按域名搜索

### 单节点详情接口

用于查看单个博客的详细信息与关联关系。

示例路径：

- `GET /api/blogs/{id}`

建议返回：

- 节点基本信息
- 该节点指向的友链
- 首次发现来源
- 最近处理日志摘要

### 边列表接口

用于获取连接关系。

示例路径：

- `GET /api/edges`

建议支持：

- 按 `from_blog_id` 查询
- 按 `to_blog_id` 查询
- 分页

### 图谱接口

用于前端图可视化。

示例路径：

- `GET /api/graph`

建议返回：

- `nodes`
- `edges`

### 统计接口

用于展示前端统计卡片。

示例路径：

- `GET /api/stats`

建议返回：

- 博客总数
- 边总数
- 平均友链数
- 最大深度
- 各状态数量分布

### 日志接口

用于查看最近运行日志。

示例路径：

- `GET /api/logs`

## 4.3 后续可选控制接口

这些接口可以预留，但不一定纳入 MVP：

- `POST /api/crawl/start`
- `POST /api/crawl/stop`
- `POST /api/crawl/resume`
- `POST /api/seeds/import`

## 4.4 API 文档方式

建议采用 `FastAPI` 自动生成：

- Swagger UI
- OpenAPI JSON

第一阶段无需额外单独维护一份复杂接口文档，但需要在 `plan.md` 中明确各接口职责和返回数据方向。

## 当前建议的阶段划分

### MVP 第一阶段

先完成：

- 种子导入
- 爬虫主流程
- SQLite 持久化
- 节点与边存储
- 中断恢复
- 基础 REST API
- 基础前端运行状态页
- 图数据导出

### 第二阶段

增强：

- 浏览器 / MCP 兜底抓取
- 图谱交互可视化
- 更强的博客识别规则
- 控制接口
- 更丰富的日志与监控

## 当前待确认问题

1. 持久化主存储是否确定使用 `SQLite`？
2. 前端第一阶段是否只做运行状态面板，不急着做复杂图谱交互？
3. 后端第一阶段是否默认只走 HTTP 抓取，浏览器 / MCP 暂时作为增强项？
4. API 第一阶段是否以只读查询接口为主，控制接口后置？
5. 是否将 `robots.txt`、限速、`User-Agent` 伪装纳入 MVP？
